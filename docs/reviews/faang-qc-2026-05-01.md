## Rejected findings (verified false positives)

The audit discipline rule "verify before claiming" caught two sub-agent Blocker claims that don't survive ground-truth checks:

### REJECTED-1 — Sub-agent 4: ".env files committed to git with real secrets"
**Claim.** `.env`, `api/.env`, `dashboard/.env.local` are committed to git history with `sk_live_...` and `cdc33e44...` exposed.

**Verification.**
```bash
git ls-files .env api/.env dashboard/.env.local  # → empty (none tracked)
git check-ignore -v .env api/.env dashboard/.env.local
# .gitignore:15:.env       .env
# api/.gitignore:11:.env   api/.env
# dashboard/.gitignore:23:.env*.local  dashboard/.env.local
```

All three files are gitignored and untracked. The sub-agent saw them on disk and assumed they were committed; it didn't run `git ls-files`. Only `.env.example` files are in the repo, with safe placeholder content.

### REJECTED-2 — Sub-agent 5: "Deploy scripts allow wrong-cwd deploys"
**Claim.** `scripts/deploy-dashboard.sh` could still ship the API image to the dashboard app if run from the wrong cwd.

**Verification.** Read [`scripts/deploy-dashboard.sh:12-31`](../scripts/deploy-dashboard.sh#L12-L31):
- Line 12-13: `SCRIPT_DIR` and `DASHBOARD_DIR` are derived from `$(dirname "$0")` — the script's actual location, completely independent of caller cwd
- Line 15-16: validates `fly.toml` exists in `$DASHBOARD_DIR` AND contains `app = "robot-dashboard-andres-morones"`
- Line 19: explicit `cd "$DASHBOARD_DIR"` before any `flyctl` call
- Line 22-31: post-deploy `/api/health` fingerprint check exits non-zero if the wrong image lands

The whole point of the design is cwd-independence. The agent's claimed risk is not reachable.

# Section B — Per-File Deep Dive

Ordered by severity-weighted finding count.

### `api/app/config.py`

**Findings:**
- [High] line 17 — Default `api_bearer_token: str = "dev-token-change-me"` shipped in source.
  - **Cites:** ADR-008 §1; `.env.example` correctly uses placeholder pattern; CLAUDE.md "Auth" — `API_BEARER_TOKEN` lives in Fly secret.
  - **Why it matters:** A reviewer could accidentally deploy with a known token if env-var injection silently fails. Placeholder default masks the failure.
  - **Suggested fix:** Default to `""`; raise in lifespan if still empty after env load.

### `dashboard/src/app/dashboard/page.tsx`

**Findings:**
- [Medium] line 41 — `export const revalidate = 0` and `export const dynamic = "force-dynamic"` together force fully dynamic, contradicting ADR-007's 30s revalidate.
  - **Cites:** ADR-007 §2 "Next.js revalidate=30 + FastAPI TTLCache(30s)".
  - **Why it matters:** Forces every render to bypass ISR; underlying fetches still use 30s, creating mixed signals about caching intent.
  - **Suggested fix:** Set `revalidate = 30`; keep the explanatory comment but update to match ADR-007.

### `dashboard/src/components/calls-table.tsx`

**Findings:**
- [High] line 40 — TODO references `docs/hr-twin-pending-register.md (HT-1)` which doesn't exist.
  - **Cites:** No matching file in `docs/`; `notes` column rendering blocked on a never-resolved blocker.
  - **Why it matters:** Inline TODO referencing a missing doc signals incomplete shipped feature. A FAANG reviewer reads this and asks "what's HT-1?"
  - **Suggested fix:** Either implement the rendering or remove the placeholder branch + TODO; document deferral in memory if relevant.

### `dashboard/src/components/quality/flagged-card.tsx`

**Findings:**
- [Medium] line 38 — Header label "flagged calls drilldown · CHS < 70" exposes internal "CHS" jargon.
  - **Cites:** Memory `feedback_no_internal_field_names_in_ui.md` (always translate snake_case + internal initialisms).
  - **Why it matters:** "CHS" is internal. End users see "Quality score" everywhere else; consistency matters.
  - **Suggested fix:** Rename to "Flagged calls · Quality score < 70".

### `api/app/services/dashboard_aggregations.py`

**Findings:**
- [High] line 112 — `date_range_clause(from_, to_, prefix=...)` always returns `""` due to Cloudflare WAF blocking `created_at` filters; signature implies SQL filtering that never happens.
  - **Cites:** Function docstring + memory `reference_cloudflare_waf_twin.md`.
  - **Why it matters:** 6+ callers pass `from_` and `to_` expecting the clause to filter at the SQL layer. They're then expected to filter Python-side. The intent isn't obvious from the call site.
  - **Suggested fix:** Rename to `_waf_blocked_date_clause()` and add a one-line comment at every callsite explaining the actual filtering happens Python-side.

### `api/app/routers/dashboard.py`

**Findings:**
- [Medium] lines 407-410 — `twin_query_audit_sample` is a named helper called from exactly one place (line 353).
  - **Cites:** Single-caller pattern; CLAUDE.md "Lean design".
  - **Why it matters:** Indirection adds a hop without abstraction value.
  - **Suggested fix:** Inline into the `quality()` endpoint, or keep the helper and document why (e.g. for testability — `tests/conftest.py:69` patches it).

### `api/app/services/event_bus.py`

**Findings:**
- [Medium] lines 51-62 — `put_nowait` silently drops events on full queue (maxsize=100); only logs a warning.
  - **Cites:** ADR-009 "Option C hybrid" — depends on event delivery for SSE freshness.
  - **Why it matters:** A slow consumer or backlog silently loses updates. SSE clients won't notice. No metric to detect drops.
  - **Suggested fix:** Add a `prometheus_client.Counter("event_bus_queue_dropped_total")` and bump it on each drop. Promote log to ERROR if drop rate > 1/min.

# Section C — ADR Drafts (stubs for user to copy into `docs/decisions/`)

### ADR-014 — Telemetry quadrant content vs locked theme spec

**Status:** Proposed
**Context:** Memory `project_dashboard_theme_composite_locked` calls for 7 Bloomberg Pit quadrants on the Telemetry tab. Current ship: 4 stable widgets (Controls, KpiStrip, RpmTpmChart, LatencyPercentilesCard) plus 2 added 2026-05-01 (TokenSpendTrendCard, CostPerBookingCard). 4 of the original 7 (#3 dead-air, #4 time-to-first-utterance, #6 per-turn sentiment, #7 transcript search) are blocked on backend changes (per-turn timestamps, search endpoint).
**Decision:** Ship the current 6 quadrants as Telemetry v1. Document the backend gaps that would unlock the remaining 4 (per-bucket TPM role-tag split in aggregator; per-turn timestamps from HR; backend search endpoint).
**Consequences:** The locked theme spec stays the north star but is acknowledged as a partial-ship. Backend work to unlock remaining quadrants is Tier-2.

### ADR-015 — Workflow snapshot persistence

**Status:** Proposed
**Context:** The HR workflow is the live source of truth for the voice-agent behavior. Only v5.1 prompt and an interim architecture map are in the repo; v5.2 prompt edits live in HR UI only. Risk: a fork or version rollback in HR loses the v5.2 improvements with no repo recovery path.
**Decision:** Before each HR workflow publish, snapshot the workflow JSON via HR REST `GET /versions/{vid}/nodes` and commit to `docs/references/happyrobot/workflow-v<N>-snapshot.json`. Memory `reference_hr_fork_corruption.md` warns HR API *mutations* corrupt forks; reads are safe.
**Consequences:** One snapshot file per shipped HR version; sized ~150KB. Recoverable from repo if HR state is lost. Manual step before publish (could be scripted in `scripts/snapshot-hr-workflow.sh`).
